# Block System — Provisioning

## Manifest (`freeloader.yaml`)

The project manifest is the single user-facing configuration file. It lists which blocks the project uses, and provides any user-overridable config values for each.

### Schema

```yaml
project:
  name: my-app            # Used as the default value for block `name` config fields.
  description: "..."      # Optional project description.
  path: /absolute/path/to/project  # Injected as `target_folder` into any block that declares it.

blocks:
  - use: gitlab/registry  # Reference to the block folder: {provider}/{block_name}
    id: registry          # Optional alias. Defaults to the value of `use`.
    config:               # Flat dict of config field overrides (only basic by default).
      name: my-app
      visibility: private

  - use: github/repo
    config:
      name: my-app

  - use: coolify/project
    config:
      name: my-app

  - use: coolify/app
    config:
      name: my-app
      domain: myapp.example.com
```

### `BlockRef` fields

| Field    | Type        | Description                                                           |
|----------|-------------|-----------------------------------------------------------------------|
| `use`    | `str`       | Block identifier: `{provider}/{block_name}`. Must match a block folder. |
| `id`     | `str`       | Optional alias. Used in DAG wiring and log output. Defaults to `use`. |
| `config` | `dict`      | User-provided overrides for basic (and optionally advanced) fields.   |

The `resolved_id` for a block is `id` if set, otherwise `use`.

## Initialization: `freeloader block add`

When a user adds a block, `freeloader block add` generates the manifest `config` section using `ConfigBuilder`. By default, only **basic** fields are emitted:

```yaml
# Generated by: freeloader block add gitlab/registry
- use: gitlab/registry
  config:
    name: my-app        # basic, required
    visibility: private # basic, has default
```

With `--full`, advanced fields are also emitted with their defaults, making them visible and overridable:

```yaml
# Generated by: freeloader block add gitlab/registry --full
- use: gitlab/registry
  config:
    name: my-app
    visibility: private
    token_scopes:       # advanced
      - read_registry
      - write_registry
```

**Secrets are never written to the manifest.** The user is prompted to store them in the vault separately (`freeloader secrets write`).

## Provisioning: `freeloader provision`

### Step 1: Load and Validate

```
freeloader.yaml  →  ProjectManifest (Pydantic)  →  list[BlockRef]
```

Each `BlockRef` is validated — `use` must resolve to a known block folder.

### Step 2: Load Block Contracts

For every `BlockRef`, load the corresponding `block.yml` and deserialise it into a `BlockContract`.

```
blocks/gitlab/registry/block.yml  →  BlockContract
blocks/github/repo/block.yml      →  BlockContract
...
```

### Step 3: DAG Resolution

`DAGResolver.resolve(block_refs, contracts)` produces an ordered `list[ResolvedBlock]`.

**The resolver:**
1. Builds a `provides_map`: `"{layer}.{output_name}" → [block_id, ...]` for every provides port of every block.
2. For each block, matches each entry in `requires` against `provides_map`.
3. Raises `MissingRequirement` if a required (non-optional) port has no provider.
4. Raises `AmbiguousProvider` if a port is provided by more than one block.
5. Records `block_id → provider_id` wiring for each resolved port.
6. Runs Kahn's algorithm (topological sort) on the dependency graph.
7. Returns blocks in execution order, each carrying its resolved input wiring.

### Step 4: Execute in Order

For each `ResolvedBlock`:

1. **Resolve secrets**: fetch secret values from the vault for fields listed under `config.secrets`.
2. **Resolve input ports**: look up the provider block's outputs in `ExecutionContext`.
3. **Build tfvars**: merge `block_ref.config` + resolved secrets + resolved port values.
4. **Run Terraform**: call `Terraform(work_dir).prepare(template, tfvars)` then `Terraform.apply()` — the `Terraform` facade from `freeloader.shared.terraform` is the only permitted abstraction for all Terraform operations. Never call `TerraformRunner`, `TerraformFile`, or `TerraformResource` directly.
5. **Capture outputs**: call `Terraform(work_dir).output()`, store result in `ExecutionContext`.

### Step 5: Done

After all blocks complete, the `ExecutionContext` holds all outputs keyed by `block_id.output_name`. The provisioner prints a summary.

## Walk-through: Full Python Deployment Pipeline

### Goal
Deploy a Python/pip web app using:
- Local file generation (Dockerfile, .gitignore, .dockerignore)
- GitLab as image registry
- GitHub as source control (with Actions secrets injected)
- AWS EC2 as compute (for future SSH access; Coolify self-hosted on it)
- Coolify to manage the project and deploy the Docker image

### Manifest

```yaml
project:
  name: acme-api
  description: ACME backend API
  path: /home/user/projects/acme-api

blocks:
  - use: docker/dockerfile
    config:
      language: python
      package_manager: pip
      server: nginx
      # target_folder injected from project.path

  - use: docker/dockerignore
    config:
      language: python

  - use: git/gitignore
    config:
      language: python

  - use: gitlab/registry
    config:
      name: acme-api
      visibility: private

  - use: coolify/project
    config:
      name: acme-api

  - use: github/repo
    config:
      name: acme-api
      org: acme-corp
      protect_main: true

  - use: coolify/app
    config:
      name: acme-api
      domain: api.acme.com
      ports_exposes: "8080"
```

### DAG Resolution

The resolver builds this dependency graph:

```
docker/dockerfile   ─── (no deps) ───────────────────┐
docker/dockerignore ─── (no deps) ───────────────────┤
git/gitignore       ─── (no deps) ───────────────────┤
gitlab/registry     ─── (no deps) ───────────────────┤
coolify/project     ─── (no deps) ───────────────────┤
                                                      ↓
coolify/app ← requires: registry.image_path   ─── gitlab/registry
            ← requires: platform.project_uuid ─── coolify/project

github/repo ← requires: registry.host         ─── gitlab/registry (optional)
            ← requires: registry.user         ─── gitlab/registry (optional)
            ← requires: registry.token        ─── gitlab/registry (optional)
            ← requires: registry.image_path   ─── gitlab/registry (optional)
            ← requires: deploy.webhook_url    ... (not connected — no provider in this pipeline)
```

After topological sort:

| Step | Block               | Inputs wired from                     |
|------|---------------------|---------------------------------------|
| 1    | `docker/dockerfile` | *(user config only)*                  |
| 2    | `docker/dockerignore` | *(user config only)*                |
| 3    | `git/gitignore`     | *(user config only)*                  |
| 4    | `gitlab/registry`   | *(user config + secrets only)*        |
| 5    | `coolify/project`   | *(user config + secrets only)*        |
| 6    | `coolify/app`       | `gitlab/registry.image_path`, `coolify/project.project_uuid` + secrets |
| 7    | `github/repo`       | `gitlab/registry.{host,user,token,image_path}` + secrets |

Steps 1–5 have no inter-block dependencies. The resolver may execute them in any order consistent with the topological sort (layer order acts as tie-breaker, local assets first).

### Result

After all 7 steps:
- Dockerfile, .gitignore, .dockerignore written to project folder.
- GitLab container registry created; access token generated.
- Coolify project created.
- Coolify app deployed from image; domain configured.
- GitHub repo created with branch protection and 4 Actions secrets pre-configured.

The developer can now clone, push code, set up the CI workflow (`github/actions_ci` block), and the pipeline is ready.

## Error Cases

| Error                | Cause                                                         | Resolution                                             |
|----------------------|---------------------------------------------------------------|--------------------------------------------------------|
| `MissingRequirement` | A required port has no provider block in the manifest.        | Add the missing block, or mark the port `optional`.    |
| `AmbiguousProvider`  | Two blocks provide the same port key.                         | Assign unique `id` aliases and update routing.         |
| `DuplicateBlockId`   | Two block entries resolve to the same `resolved_id`.          | Set a unique `id` alias on one of them.                |
| `CircularDependency` | Blocks form a cycle in their requires graph.                  | Re-evaluate block boundaries; split the cycle.         |
| Terraform error      | Provider auth failure, quota, network.                        | Shown inline; fix config/secrets and re-run.           |
